Replace the current read-then-write `UpsertAsync` method with an atomic PostgreSQL `INSERT ... ON CONFLICT DO UPDATE` via EF Core's `ExecuteSqlAsync` with a `FormattableString`.

The new implementation:

1. Remove the existing `UpsertAsync` method body (lines ~124-178 in current code).

2. Replace with a single `ExecuteSqlAsync` call using `FormattableString` (NOT `ExecuteSqlRawAsync` — use `ExecuteSqlAsync` which accepts `FormattableString` and auto-parameterizes via `$""`).

3. The SQL must:
   - INSERT into `telegram_users` with all fields needed for a new user row: `telegram_user_id`, `username`, `first_name`, `last_name`, `user_photo_path`, `photo_hash`, `is_active`, `is_trusted`, `is_bot`, `is_banned`, `bot_dm_enabled`, `first_seen_at`, `last_seen_at`, `created_at`, `updated_at`.
   - ON CONFLICT on `telegram_user_id` (the PK), DO UPDATE SET only: `username`, `first_name`, `last_name`, `user_photo_path`, `photo_hash`, `is_active = true` (hardcoded — sending a message makes user active), `last_seen_at`, `updated_at`.
   - Do NOT update `is_trusted`, `bot_dm_enabled`, `is_banned`, `ban_expires_at`, `banned_at`, `kick_count`, `warnings`, `first_seen_at`, `created_at`, or any profile scan fields.
   - For the INSERT part: set `is_trusted` based on `TelegramConstants.IsSystemUser(user.TelegramUserId)`, set `is_bot` from `user.IsBot`, set `is_banned = false`, `bot_dm_enabled = false`, `is_active = false` (new users start inactive), `first_seen_at = now`, `created_at = now`.

4. Use `DateTimeOffset.UtcNow` for `updated_at` (and `created_at`/`first_seen_at` for INSERT).

5. Keep the logging after the SQL call. Use a simple debug log: "Upserted Telegram user {TelegramUserId} (@{Username})". The SQL operation is atomic — we cannot distinguish insert from update without `RETURNING`, but that level of detail is not needed. Remove the separate insert vs update log branches.

6. The method signature stays the same: `public async Task UpsertAsync(UiModels.TelegramUser user, CancellationToken cancellationToken = default)`.

Example shape (executor should refine):
```csharp
public async Task UpsertAsync(UiModels.TelegramUser user, CancellationToken cancellationToken = default)
{
    await using var context = await _contextFactory.CreateDbContextAsync(cancellationToken);
    var now = DateTimeOffset.UtcNow;
    var isTrusted = TelegramConstants.IsSystemUser(user.TelegramUserId);

    await context.Database.ExecuteSqlAsync($"""
        INSERT INTO telegram_users (
            telegram_user_id, username, first_name, last_name,
            user_photo_path, photo_hash, is_active, is_trusted,
            is_bot, is_banned, bot_dm_enabled,
            first_seen_at, last_seen_at, created_at, updated_at
        ) VALUES (
            {user.TelegramUserId}, {user.Username}, {user.FirstName}, {user.LastName},
            {user.UserPhotoPath}, {user.PhotoHash}, {false}, {isTrusted},
            {user.IsBot}, {false}, {false},
            {now}, {user.LastSeenAt}, {now}, {now}
        )
        ON CONFLICT (telegram_user_id) DO UPDATE SET
            username = EXCLUDED.username,
            first_name = EXCLUDED.first_name,
            last_name = EXCLUDED.last_name,
            user_photo_path = EXCLUDED.user_photo_path,
            photo_hash = EXCLUDED.photo_hash,
            is_active = true,
            last_seen_at = EXCLUDED.last_seen_at,
            updated_at = {now}
        """, cancellationToken);

    _logger.LogDebug(
        "Upserted Telegram user {TelegramUserId} (@{Username})",
        user.TelegramUserId,
        user.Username);
}
```

IMPORTANT: The `$"""..."""` raw string literal with FormattableString ensures all `{...}` expressions become parameterized SQL parameters. This is NOT string interpolation into SQL — EF Core's `ExecuteSqlAsync(FormattableString)` overload handles parameterization.
