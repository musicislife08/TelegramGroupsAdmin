<feature>
  <name>ClamAV env var override in CreateClamClientAsync + GetHealthAsync</name>
  <files>TelegramGroupsAdmin.ContentDetection/Services/ClamAVScannerService.cs, TelegramGroupsAdmin.UnitTests/ContentDetection/ClamAVScannerServiceTests.cs</files>
  <behavior>
    CLAM-01: When both CLAMAV_HOST="shared-clam" and CLAMAV_PORT="3311" are set, CreateClamClientAsync constructs ClamClient("shared-clam", 3311) without calling GetConfigAsync (no DB read).
    CLAM-02a: When only CLAMAV_HOST is set (CLAMAV_PORT missing/null), DB config is used.
    CLAM-02b: When only CLAMAV_PORT is set (CLAMAV_HOST missing/null), DB config is used.
    CLAM-02c: When CLAMAV_PORT is set to a non-integer string "notanumber", DB config is used.
    CLAM-02d: When CLAMAV_HOST is empty string and CLAMAV_PORT is set, DB config is used.
    CLAM-02e: When neither env var is set, DB config is used.
    CLAM-03a: First call with both env vars set produces exactly one INFO log containing "ClamAV env var override active" and the effective host:port.
    CLAM-03b: Second call with both env vars set produces NO additional INFO log (logged once per process, not per scan).
    CLAM-04a: GetHealthAsync returns FileScannerHealthResult with Host/Port matching env var values when override is active.
    CLAM-04b: GetHealthAsync log messages show override host:port (not DB host:port) when override is active.
  </behavior>
  <implementation>
    1. Add `private static volatile bool _hasLoggedOverride;` field to ClamAVScannerService (static because service is Scoped — flag must persist across request scopes for "log once per process" semantic).

    2. Add `GetEffectiveEndpointAsync` private method returning `(string host, int port)`:
       - Read `CLAMAV_HOST` and `CLAMAV_PORT` via `Environment.GetEnvironmentVariable`.
       - If both are non-null/non-whitespace AND `int.TryParse(envPort, out var parsedPort)` succeeds:
         - If `!_hasLoggedOverride`, set `_hasLoggedOverride = true` and log INFO: "ClamAV env var override active -- using {Host}:{Port} (CLAMAV_HOST / CLAMAV_PORT)".
         - Return `(envHost, parsedPort)`.
       - Otherwise: call `GetConfigAsync(cancellationToken)` and return `(config.Tier1.ClamAV.Host, config.Tier1.ClamAV.Port)`.

    3. Refactor `CreateClamClientAsync` to call `GetEffectiveEndpointAsync` and construct `new ClamClient(host, port)`.

    4. Refactor `GetHealthAsync` to call `GetEffectiveEndpointAsync` at the top and use returned `(host, port)` for ALL log messages, error messages, and the `FileScannerHealthResult.Host`/`.Port` fields. Remove the separate `GetConfigAsync` call at the top of GetHealthAsync — it is no longer needed for host/port (Enabled check is only in ScanFileAsync, not GetHealthAsync).

    5. Fix `ScanFileAsync` line 122-123: the error log for ping failure shows `config.Tier1.ClamAV.Host/Port` but the client was constructed with possibly-overridden values. Since `ScanFileAsync` already calls `GetConfigAsync` for the Enabled check (line 61), capture effective endpoint separately. The simplest fix: call `GetEffectiveEndpointAsync` once in the retry loop's ping-failure log (or before the loop, storing the result).

    IMPORTANT notes:
    - Use `string.IsNullOrWhiteSpace` (not just `!= null`) per research pitfall 3 — container orchestrators can set empty strings.
    - Do NOT cache env var values at startup — read per-call per CLAM-01.
    - Do NOT mutate config.Tier1.ClamAV.Host/Port — the FileScanningConfig is a shared DTO.
    - The static _hasLoggedOverride field does NOT need a reset mechanism in production. Tests MUST reset it via reflection in [TearDown] to avoid cross-test pollution (or use a `[OneTimeSetUp]` approach).

    Test approach (NUnit + NSubstitute):
    - ClamAVScannerService constructs ClamClient internally (private method, no DI for ClamClient). Tests cannot directly verify what host:port was passed to ClamClient constructor.
    - Strategy: Since ClamClient is constructed with host/port and `ScanFileAsync`/`GetHealthAsync` will fail to connect in tests (no clamd running), test the observable effects:
      - Mock ISystemConfigRepository to verify it IS or IS NOT called (verifying override skips DB).
      - Use a real ILogger (or NSubstitute mock) to assert INFO log was emitted with correct host:port.
      - For GetHealthAsync: verify the returned FileScannerHealthResult.Host and .Port match env var values (this is directly observable).
    - Alternative simpler strategy: Since `GetEffectiveEndpointAsync` is private, expose it as `internal` with `[InternalsVisibleTo("TelegramGroupsAdmin.UnitTests")]` — but this requires modifying the .csproj. The observable-effects approach is preferred to avoid coupling tests to private API.
    - Env var setup in tests: `Environment.SetEnvironmentVariable("CLAMAV_HOST", value)` in test, clear in `[TearDown]`.
    - Static field reset: Use reflection to reset `_hasLoggedOverride` in `[TearDown]`: `typeof(ClamAVScannerService).GetField("_hasLoggedOverride", BindingFlags.NonPublic | BindingFlags.Static)!.SetValue(null, false);`
    - For CLAM-01 (override skips DB): set both env vars, mock `_configRepository.GetAsync` to throw, call a method that exercises CreateClamClientAsync. If override works, the mock is never hit and no exception is thrown. For GetHealthAsync specifically, the method will still fail on the ClamClient.PingAsync (no daemon), but the returned FileScannerHealthResult shows the env var host/port.
    - For CLAM-02 (fallback to DB): set only one env var, mock `_configRepository.GetAsync` to return a config with known DB values. GetHealthAsync will fail on ping but return error with DB host:port.
    - For CLAM-03 (log once): use NSubstitute mock logger. After first call, assert LogInformation was called once with the override message. After second call, assert still only called once (count does not increase).
    - For CLAM-04 (GetHealthAsync uses override): set both env vars, call GetHealthAsync. The returned result's Host/Port must match env var values.
