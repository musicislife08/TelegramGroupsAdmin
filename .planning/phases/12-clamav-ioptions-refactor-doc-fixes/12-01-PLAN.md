<objective>
Refactor ClamAV env var override from raw `Environment.GetEnvironmentVariable("CLAMAV_HOST"/"CLAMAV_PORT")` to `IOptions<ClamAVConfig>` bound via ASP.NET Core configuration. This fixes the broken override (compose sets `CLAMAV__HOST` double-underscore but code reads `CLAMAV_HOST` single-underscore) and aligns with ASP.NET Core conventions already used elsewhere in the project (`AppOptions`, `ContentDetectionOptions`).

Purpose: CLAM-01 requires IOptions<ClamAVConfig> binding. The current implementation silently fails in Docker Compose deployments because the env var name mismatch means the override never activates.

Output: ClamAVScannerService uses IOptions injection, ClamAVConfig has nullable-friendly defaults for override detection, SystemConfigRepository merge logic updated, all 9 existing tests pass with IOptions mocks, production compose includes ENABLE_METRICS.

**Step 2 — Fix SystemConfigRepository.MergeConfigs sentinel logic:**

The `MergeConfigs` method (around line 659) currently uses `new ClamAVConfig().Host` and `new ClamAVConfig().Port` as sentinels to detect whether a chat-level config has overridden the default. After changing defaults, `new ClamAVConfig().Host` is `null` and `new ClamAVConfig().Port` is `0`.

The merge logic currently checks: "If chat config value differs from class default, use chat's value; otherwise use global's value." With nullable Host and zero Port, the check becomes:
- `chatConfig.Tier1.ClamAV.Host != null` — if chat has ANY non-null host, it's considered an override. This is correct because the database stores non-null values for configured chats and null when not customized.
- `chatConfig.Tier1.ClamAV.Port != 0` — if chat has ANY non-zero port, it's considered an override. Same reasoning.

However, verify the database serialization: JSONB deserialization of a chat config that has NOT customized ClamAV Host will deserialize with the class default (null). So the sentinel comparison `!= null` / `!= 0` is actually correct.

Update the merge logic to use the new sentinel values. Replace the three `new ClamAVConfig()` allocations with a single `static readonly` field for clarity:

```csharp
// At class level or method level:
// The sentinel is now simply the default-constructed ClamAVConfig (Host=null, Port=0, TimeoutSeconds=30)
// No code change needed IF the comparisons still work. But verify:
// - chatConfig.Tier1.ClamAV.Host != null → chat has customized host → use chat's
// - chatConfig.Tier1.ClamAV.Port != 0 → chat has customized port → use chat's
// - chatConfig.Tier1.ClamAV.TimeoutSeconds != 30 → unchanged sentinel
```

The `new ClamAVConfig()` pattern still works because the sentinel values changed in lockstep with the class. However, for readability and to avoid repeated allocations, extract a `private static readonly ClamAVConfig DefaultClamAV = new();` and use it. This is a style improvement, not a behavior change.
