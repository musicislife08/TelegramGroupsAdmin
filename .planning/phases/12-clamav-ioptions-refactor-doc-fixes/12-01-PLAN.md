**CRITICAL CONTEXT for MergeConfigs sentinel decision (pre-analyzed from SystemConfigRepository.cs):**

`SaveAsync` (line 72) serializes the ENTIRE `FileScanningConfig` object with `JsonSerializer.Serialize(config, _jsonOptions)`. When a chat config is saved for ANY reason (e.g., to toggle VirusTotal), the full nested object graph is serialized, including `Tier1.ClamAV` with whatever values the class defaults produced at the time. This means existing database rows WILL contain explicit `"host":"localhost"` and `"port":3310` values in their JSONB for chats that never customized ClamAV — these are the historical class defaults baked into the stored JSON.

After changing ClamAVConfig defaults to `null`/`0`, `new ClamAVConfig()` would produce `null`/`0`. But the database still stores `"localhost"`/`3310`. The comparison `chatConfig.Tier1.ClamAV.Host != new ClamAVConfig().Host` would become `"localhost" != null` = `true`, incorrectly treating the stored historical default as a chat-level override.

**Therefore: MergeConfigs MUST use hardcoded historical sentinel constants, NOT `new ClamAVConfig()`.**
